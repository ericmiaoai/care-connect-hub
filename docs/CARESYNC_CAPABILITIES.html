CareSync v1.2

Capabilities & Technical Overview  ·  CIMA 130 — Introduction to AI  ·  May 2026

CareSync is a Progressive Web App (PWA) that helps families and professional caregivers coordinate the care of a patient — tracking medications, appointments, daily tasks, and team communication in one shared, real-time workspace. The core AI feature, Scan AVS, uses Google's Gemini 2.5 Flash multimodal model to extract structured data from After-Visit Summary documents, eliminating manual data entry while keeping the caregiver in full control of what gets saved.

1. Care Circle Model — Roles & Permissions

Every piece of data in CareSync belongs to exactly one Care Circle — a secure, bounded workspace shared by a patient's care team. Data from one circle is never visible to members of another. Access is controlled by three roles enforced at the database level via PostgreSQL Row Level Security.

🔒 Enforced at the database layer
Every row in every table is protected by PostgreSQL Row Level Security (RLS). Even a direct API call bypassing the app UI cannot read or write data outside a user's care circle. The role hierarchy is the single source of truth.

5. Scan AVS — AI-Powered Document Processing

🤖 AI Model: Google Gemini 2.5 Flash
Gemini 2.5 Flash is a multimodal large language model capable of reading both images and text. CareSync sends a compressed image of a medical After-Visit Summary and asks Gemini to extract structured care data — no manual typing required. The model name is controlled by an environment variable and can be updated without code changes when Google releases new versions.

What is an After-Visit Summary (AVS)?

An AVS is a printed document given to patients at the end of a medical visit. It contains medication changes, follow-up appointments, care instructions, and the provider's name. Manually entering this data into a care coordination app is time-consuming and error-prone. Scan AVS eliminates this step.

Data contract — what Gemini returns

FieldTypeDescription
patient_namestringFull name extracted from document header
visit_dateISO 8601 dateDate of the medical visit
provider_namestringPhysician or clinic name
medicationsarrayMedication name, dosage, frequency, instructions
appointmentsarrayFollow-up appointment title, date/time, location
instructionsarrayCare instructions and notes
🔒 PHI never enters the database
The extracted data (medications, instructions) is stored only in the device's local IndexedDB during the HITL review. Only the caregiver's approval decision and non-sensitive metadata (visit date, provider name) are written to Supabase. This is a deliberate architectural choice to minimise Protected Health Information exposure.

Environment variable control

The AI model name is not hardcoded. It is read from the GEMINI_MODEL environment variable at runtime. When Google releases a new Gemini version, only the environment variable needs updating — no code deployment required.

8. Status Indicators & User Notifications

CareSync displays two amber status banners at the top of the app to inform users about connectivity and synchronization state:

📡 "You're offline" Banner

Appears when the browser detects no network connection (uses the browser's navigator.onLine API and online/offline events).

When: User loses internet connectivity
Icon: WiFi Off (lucide-react)
Message: "You're offline — changes are paused until connectivity is restored."
Behavior: New tasks, edits, and real-time sync are paused. The app displays cached data but doesn't send any changes to the server.
Recovery: Disappears automatically when connectivity returns.

🔄 "Live sync interrupted" Banner

Appears when one or more Supabase realtime WebSocket subscriptions fail to connect or time out. This is a server-level issue, distinct from browser offline.

When: Supabase realtime subscription enters CHANNEL_ERROR or TIMED_OUT state (happens ~40–45 seconds after a network disruption or Supabase service issue)
Icon: Refresh Circular (lucide-react)
Message: "Live sync interrupted — data may be delayed. Refresh the page to reconnect."
Behavior: Other users' real-time changes (new tasks, appointments, broadcasts, member updates) are not synced to this user's view until the subscription recovers.
Recovery: Subscriptions auto-reconnect within 5–7 seconds in most cases, or user can manually refresh the page.
Technical note: Each of 7 realtime channels is independently monitored (tasks, calendar events, broadcasts, members, care circle state, membership changes, new member alerts). If any single channel fails, the banner appears.
⚠️ Two types of connectivity
Browser offline (no internet) is different from realtime sync interrupted (server unreachable via WebSocket). Both are shown to help users understand what went wrong and what action to take.

Realtime Sync

Four tables are enrolled in Supabase's Postgres Changes publication. When any member creates, updates, or deletes a task or appointment, every other online member's view updates within 200–500 ms via WebSocket — no page refresh needed. Role changes also propagate in real time.

10. Security & Privacy Design

Row Level Security (RLS)

Every table in the database has RLS enabled. All policies are anchored to a single helper function, is_circle_member(), which checks whether the requesting user belongs to the target care circle with an appropriate role. A user who does not belong to a circle receives zero rows from any query — not an error, just empty results.

Human-in-the-Loop (HITL) pattern

The Scan AVS feature implements a deliberate safety gate: AI output is never committed to the database without explicit human approval. This addresses two risks inherent in automated document parsing:

  • Hallucination — the model might extract incorrect data from a blurry image
  • Wrong patient — a mismatched document triggers a visible orange warning banner before the user can approve

PHI Containment

Protected Health Information never enters the server database
Medications, care instructions, and clinical details extracted by Gemini are stored exclusively in device-local IndexedDB during the review window. Only after human approval are care tasks and appointments written to Supabase — and those records contain task titles and dates, not raw clinical text.

Scan AVS — AI Pipeline (Swim Lane)

Each column represents one actor. Highlighted cells show which actor is active at that step. Inactive cells are shown in gray.

👤 User (Caregiver) ⚙️ CareSync 🤖 AI & Cloud (Gemini)
Photographs or uploads the AVS document from their device. — —
— Validates file type; compresses image to JPEG (max 1600 px, ~1 MB). Attaches the user's Supabase JWT to authenticate the request. —
— Sends compressed image + structured extraction prompt to the Netlify cloud function. Gemini 2.5 Flash reads the image and returns structured JSON: patient name, visit date, provider, medications, appointments, and care instructions.
— Validates the JSON response against a strict schema. Checks extracted patient name against the Care Circle's patient record. —
Sees an orange warning banner if the extracted patient name does not match the care circle's patient. Surfaces the mismatch warning; does not block review — the caregiver decides whether to proceed. —
⚠️ HUMAN-IN-THE-LOOP — Inspects every extracted field; edits any errors before making a decision. Presents all fields in an editable review form. No data has been written to the database yet. —
✓ Taps Approve Creates tasks and appointments in Supabase. Saves a non-PHI audit record. All PHI remains on-device only. —
✗ Taps Reject Discards all extracted data. Nothing is written to the database. —
🔑 Key multimodal design principle
The AI model never writes directly to the database. Every Scan AVS session passes through a mandatory human review step before any data is persisted — protecting against hallucinations, wrong-patient errors, and misread documents.
